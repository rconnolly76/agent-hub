# Top 5 Recommendations Contract (v1.0)

This contract standardizes the priority recommendations generated by every skill run. It is the **structured serialization** of the recommendations that also appear in the skill's markdown report — the two must stay in lockstep.

## File name and placement

- Report skills: `_top-5-recommendations.json` in the repo root.
- Content-bundle skills: `_top-5-recommendations.json` inside the bundle directory (for example `docs/_top-5-recommendations.json`).

The push script (`agent-hub-push.mjs`) uploads the payload as the `topRecommendations` multipart field (JSON **text**, not a file blob). If missing, the server derives one from the markdown report.

## Schema

```json
{
  "version": "1.0",
  "skillType": "ux-journey-reviewer",
  "generatedAt": "2026-04-15T10:00:00.000Z",
  "recommendations": [
    {
      "priority": "P1",
      "title": "Resolve critical navigation failures",
      "action": "Fix broken nav states on the run detail page and verify with a rerun.",
      "rationale": "Critical path failure blocks user progress.",
      "severity": "critical",
      "effort": "M",
      "impact": "L",
      "expectedOutcome": "Users can navigate the run detail flow end-to-end without error.",
      "successMetric": "0 critical findings in the next ux-journey-reviewer run across all primary journeys.",
      "relatedFindings": ["JR-001", "JR-007"],
      "affectedFiles": ["src/app/runs/[id]/page.tsx"]
    }
  ]
}
```

## Rules

- `version` must be `"1.0"`.
- `recommendations` must contain 1–5 entries (ingest stores the first five ordered by priority).
- Priorities are `P1` through `P5` — unique, in order.

### Required per recommendation

| Field | Type | Notes |
|---|---|---|
| `priority` | `"P1" \| "P2" \| "P3" \| "P4" \| "P5"` | Unique within the payload. Ingest sorts ascending. |
| `title` | string | Imperative, < 100 chars. Same title the markdown report uses. |
| `action` | string | The concrete change. 1–3 sentences. |

### Optional per recommendation

| Field | Type | Notes |
|---|---|---|
| `rationale` | string | Why this matters. 1–2 sentences. |
| `severity` | `"critical" \| "warning" \| "info" \| "low" \| "investigate"` | Mirrors finding severity. |
| `effort` | `"S" \| "M" \| "L"` | S = < 1 day, M = 1–5 days, L = > 1 week or cross-team. |
| `impact` | `"S" \| "M" \| "L"` | S = incremental, M = broad, L = step-change. |
| `expectedOutcome` | string | One sentence: what changes once the action is done. |
| `successMetric` | string | Observable, measurable completion signal. |
| `relatedFindings` | string[] | Finding ids this recommendation addresses (match ids used in the markdown and in the findings table). |
| `affectedFiles` | string[] | Primary file paths the change touches. |

Optional fields are preserved in `runs.rawMetadata.topRecommendations` when present. Skills that produce richer data (effort/impact quadrant, success metric) should emit the extended fields; skills that don't may omit them. The server never rejects a payload for missing optional fields.

## Severity vs. priority

`priority` (P1–P5) is the recommendation's execution ordering — strictly a position in the ranked list. `severity` is the attached risk level drawn from the finding pool. They usually correlate (P1 often critical) but they're distinct axes: an `info`-severity finding can still be P1 if it's the highest-leverage cleanup of the run.

## Ingest behavior

- If a valid payload is provided, it is stored in `runs.rawMetadata.topRecommendations`.
- If missing or invalid, ingest derives top recommendations from findings/metrics via `buildTopRecommendationsFromFindings` (falls back to markdown parsing via `buildTopRecommendationsFromMarkdown`).
- The executive summary normalizer uses these recommendations first and always appends a prioritized top-5 block.

## Authoring discipline

The markdown report's "Top Recommendations" section is the human-readable render of this payload. Every entry in the sidecar must have a matching block in the markdown, and vice versa. If a skill uses the richer priority-quadrant/effort/impact model in its markdown, it should serialize those into the optional sidecar fields rather than dropping them — richer data flows through to the Hub UI over time.
